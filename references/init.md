# INIT / DEFINE — register the tracked systems, map each one

This is **DEFINE** mode: stand up — and keep current — the per-project home for the skill systems we steward.
A project may track **many** skill systems; DEFINE records them in a **registry** and maps each one. Two layers:

1. **The registry** — `<repo>/.agents/tracked-systems.md`: a lean index of every tracked system in this repo,
   answering *"what do we track, and where does each stand?"* (the surface **OBSERVE** reads). The registry
   **never copies the iteration log** — that is git (the `skillsys(<id>)` history); the registry only *points*
   at it. Most registry rot is duplicating into a file what `git log` already holds; don't.
2. **Per system, a map + a criteria fixture** — the free-form, notes-friendly inventory that answers *"what
   IS this system, and what workflow orchestrates it?"*, plus its per-node judging bar. Records **everything**
   about that system, with **no scores and no rigid schema**.

**Register a system** = add its registry row + build its map + seed its criteria. **Refresh** = keep all three
true as the system's shape changes. A repo with one tracked system still gets a registry (a one-row index) — it
is the stable place OBSERVE looks and the place the next system is added without re-deciding the layout.

## The registry — `<repo>/.agents/tracked-systems.md`
A lean index. One entry per tracked system; keep each entry to pointers + the smallest standing status, never a
log. The `id` is the **commit-scope key** — every optimization commit for this system is `skillsys(<id>): …`,
so the registry is what maps a commit's scope back to the system it changed.

Per system, record exactly:
- **id** — stable name, also the `skillsys(<id>)` scope key.
- **location** — repo/path of the skill set **and** its orchestrator/workflow file(s).
- **does** — its job in one line: input → output (the artifact it produces).
- **map** — pointer to this system's composition map (the per-system inventory below).
- **criteria** — pointer to its per-node criteria fixture (the judging bar; item 10).
- **judge** — the independent judge that scores its output: which model/config + where it's defined
  (`node-validation-loop.md`'s fresh judge), plus the **skill cap** and *retire-the-lowest-contribution-when-
  adding-would-exceed-it* rule (the Library-Drift Ratchet — one line; it keeps the library from drifting/bloating).
- **log** — the iteration-log pointer, verbatim: `git log --grep '^skillsys(<id>)'` (+ `scripts/review-edits.sh`).
  This **is** the record of every mistake/fix/optimization — git, not a file here.
- **open threads** — a *tiny* capped block (≤~5 lines): only **unresolved or recurring** patterns worth
  carrying forward. An absorbed lesson drops out of this block and lives on in git alone. Regenerated by
  CONSOLIDATE (the out-of-band "dream"), never grown per-edit into a ledger.

Keep the registry honest with the **exclusion list**: never write into it (or any standing `.agents/` file)
anything that is (a) recoverable from `git log`/`git blame` — that's the `skillsys` trailers; (b) in-progress
run state — that's `status.md`, outside Hermes; (c) a one-off instance rather than a generalizing pattern; or
(d) deducible from the code/skills in 5 seconds.

One entry, in full — the shape to copy (pointers + the smallest live status, never a log):
```markdown
## lesson-build
- **location:** `.claude/workflows/lesson-build.js` (orchestrator) + `skills/lesson-*` + `pi-runner/`.
- **does:** an approved lesson topic → a rendered teaching MP4 (Remotion, via the pi executor).
- **map:** `.agents/lesson-build-map.md` · **criteria:** `.agents/lesson-build-criteria.md`
- **judge:** node-validation-loop independent judge (capable model, fresh context); cap 12 skills, retire lowest-contribution on overflow.
- **log:** `git log --grep '^skillsys(lesson-build)'` (+ `scripts/review-edits.sh`)
- **open threads:** split-screen reads small at 1080p on dense beats (recurred 2×) — watching whether the visual-contract cap fixes it.
```

## What to record in a system's map
1. **Orchestrator(s) / workflow file(s).** The file that drives the run (e.g. `.claude/workflows/lesson-build.js`) and any production executor (e.g. `pi-runner/`). Note its phases/waves in order, and the shared discipline preamble it injects into every node.
2. **Nodes / waves / subagents.** Each step the orchestrator spawns, in order, with a one-line job.
3. **The wiring — which node relies on what.** For each node: the skill(s) it reads, the docs/registry it consults, the artifacts it writes. This is the most valuable column; it's how routing finds the owner.
4. **The skills**, grouped by where they're owned (this repo, a shared kit, a global skill). One line each.
5. **The governing constitution** (e.g. `CLAUDE.md`) and **architecture docs** — they are owners too (laws and facts often belong here, not in a wave skill).
6. **The capability/asset registry** if there is one (e.g. `.agents/CAPABILITIES.md` + a generated catalog) — note it's drift-gated/generated where applicable.
7. **Responsibilities — what is responsible for what.** One crisp line per owner (node, skill, doc, the orchestrator). This is the column the diagnostic loop reads to assign a problem to an owner — keep it sharp.
8. **Runtime observability — where a run leaves evidence.** This is what makes diagnosis read *real data* instead of guessing. Record every place a run writes: per-node logs, the run-status file, the raw agent transcripts, the aggregated structured returns/findings, and the product artifacts (renders, contact sheets, manifests, gate reports, verification docs) — both the dev executor and any production executor. At minimum, **every intermediate artifact + log path** so a diagnosis always has context to pull.
9. **No diagnostics ledger in the map — git is the iteration log.** Do **not** grow an append-only "what past
   diagnoses concluded" section here; that duplicates what every `skillsys(<id>)` commit already records
   (Why / Lesson / Rejected / Verify trailers), and an in-file ledger is the textbook rot that bloats the map.
   The map *gets more certain with every run* through three channels instead: **git** (the full, queryable
   record — `git log --grep '^skillsys(<id>)'`), the registry's **open-threads** block (only the unresolved /
   recurring patterns, capped, regenerated by CONSOLIDATE), and the **criteria fixture** (item 10), sharpened
   each run. If a repo already has a post-mortem/fix-log doc, that's a seed for the *first* `skillsys` commits +
   the open-threads block — fold it in and stop maintaining it as a growing file.

10. **Per-node output expectations (acceptance criteria) — a SEPARATE fixture file alongside this map** (e.g. `<repo>/.agents/skill-system-criteria.md`), one entry per producing node, (re)generatable by a per-node criteria-drafting workflow — typically seeded when the workflow's harness is created (the `transform-workflow-to-pi` adoption step) and MAINTAINED by this loop (seed it here if a repo adopted hermes without it). For each producing node, a short STANDING rubric of what a *good* output artifact looks like — the human-judged quality bar, distinct from the mechanical Output Contract (which only checks the artifact EXISTS in its lane). 3–8 observable expectations per node ("`pedagogy.md` names the whole a relation decomposes and plans complete utterances"; "the rendered MP4's teaching object fills the frame and a split reads at a glance"), each phrased to GENERALIZE across all future lessons (never hard-code the one in front of you), plus the node's known failure modes as red flags. This is BOTH the improvement target (sharpen it as the system matures) AND the **reference the human's eye judges every future run against**. It is NOT an auto-graded test (law 3) — expectations a human checks, never a reward-hackable checklist. **It is a JUDGING FIXTURE, NOT a node input: NEVER inject these criteria into the producing node's prompt.** Doing so makes the executor teach-to-the-test (it optimizes to the rubric, not to the skill) and voids the clean-room signal that tells you whether the SKILL ITSELF — not the rubric — produces good output. The criteria live in THAT fixture file (not embedded in this map, and not inside any node prompt), are read by the human/steward to judge a run and to aim the next skill edit, and never travel into the run that is being judged. Seed from each node's skill ("Wave output" / audit sections) + the real artifacts on disk; sharpen it every run (its own `skillsys(<id>)` commits are its history — there is no separate in-file ledger).

## How to gather it (no guessing)
- Read the orchestrator file: its `meta.phases`, the node prompts, and any `SK.*`/skill-path map give you nodes→skills directly.
- Read the constitution (`CLAUDE.md`) "Skills" / "Skill ownership" sections for the canonical home of each skill.
- `ls` the skills dir(s) and the docs dir.
- Where a skill is owned in another repo/kit, record the owning location (don't copy it in).

## Format
Plain Markdown. Sections + tables or bullets, whatever carries it. **Notes are encouraged** ("this node also owns the intro card", "mechanical, no skill"). **Do not** add precedence numbers, status scores, or a fixed JSON shape — the moment it becomes a strict data model it stops being honest and starts being gamed. If you want to flag something, write a note.

## Refresh
Re-run when a system's shape changes (a node added, a skill split, the orchestrator re-wired) — refresh that system's map. Add a **new** tracked system as a new registry row + its own map + criteria. The map is cheap; keeping it true is what makes routing (OPERATE step 2) reliable. A stale map is the one real failure mode here — if routing ever feels wrong, refresh the map first. The registry itself stays lean: rows are pointers, status is git + the open-threads block, never a grown ledger.

## Emit the trackers + the ambient hook (first-time setup)
DEFINE is not just the map. When you stand the Hermesian system up in a project, leave the project **aware of the pattern** so feedback auto-triggers it without anyone re-explaining:
1. **Write the registry + the map(s).** Create `<repo>/.agents/tracked-systems.md` (the index — one row per tracked system) and, for each system, its map + criteria fixture (above). The registry is the stable entry point; adding the next system is one new row + its map, no re-deciding the layout.
2. **Inject the ambient hook** into the project's agent constitution (`CLAUDE.md`, or `AGENTS.md`) — a short "Skill-system stewardship" section so every session knows the pattern is live and treats **user feedback on an artifact as a stewardship trigger**. Minimal template:
   > **## Skill-system stewardship** — We continuously evolve our tracked skill systems at dev speed. Registry = `.agents/tracked-systems.md` (what we track + where each stands; OBSERVE reads it). Treat any flaw, recurring finding, or user feedback on an artifact as a trigger: run the `hermes-skill-system` skill to capture→route→edit→verify→commit a durable, *generalizing* fix. Prefer fixing the workflow over a single skill; one `skillsys(<id>):` commit per lesson (`<id>` = the registry id); the iteration log is git, not a file; the human is the eye for visuals.
3. **Install the convention:** the `skillsys(<id>): <rule>` commit format + `scripts/review-edits.sh` span review. No new infra — git is the tracker and the iteration log.

Keep all three concise. The whole point of the hook: nobody should ever have to say "remember to update the skills" — the project already knows, so a single line of feedback is enough to invoke the right pattern.
