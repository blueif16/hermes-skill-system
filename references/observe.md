# OBSERVE — passive status read, drives no change

Answer one question: **"what skill systems do we track here, and where does each stand?"** OBSERVE is the
low-touch, always-available mode — you run it to orient before acting, to report status, or to notice a
system that has gone quiet or keeps re-breaking. It **reads only**; it never edits, commits, or reruns.
The moment OBSERVE surfaces a flaw worth fixing, that is the trigger to switch to **OPTIMIZE** (`operate.md`),
which carries the human gate — OBSERVE itself stays hands-off.

## What it reads — the index first, never the whole tree
Status lives in three places, read cheapest-first. **Do not open every system's map + criteria to answer a
status question** — that is the "load the entire docs tree like a maniac" failure mode, and it buries the
signal. Read the index; pull a single system's map only when you are about to act on it.
1. **Native memory (the router).** The orchestrator's auto-recalled notes surface *which* tracked system a
   question is about and any live concern — start here, it points you at the right entry.
2. **The registry** (`<repo>/.agents/tracked-systems.md`) — the index of tracked systems. One read gives you,
   per system: what it does, where it lives, its map/criteria/judge pointers, and its **open-threads** block
   (the only standing status text — unresolved / recurring patterns, kept tiny).
3. **git — the iteration log** (the registry's per-system `log:` pointer). The full record of every
   mistake/fix/optimization is the `skillsys(<id>)` history, queried on demand — never duplicated into a file.

## The git recipes (the "where does it stand" read path)
```bash
# Everything that changed across all tracked systems in a span (grouped, with the why-line)
scripts/review-edits.sh <since> <until>
# One system's evolution, newest-first, with each Lesson inline
git log --grep '^skillsys(<id>)' --pretty=format:'%h %ad %s%n  └ %(trailers:key=Lesson,valueonly=true)' --date=short
# What we ALREADY tried and rejected on a system (read before re-proposing — the anti-repeat read)
git log --grep '^skillsys(<id>)' --pretty='%h %s%n%(trailers:key=Rejected,valueonly=true)'
# When an invariant entered / how a rule block evolved
git log -S '<invariant phrase>' -p
```

## What OBSERVE produces
A status answer, not an artifact and not a change. Per system asked about (or all of them): **what it does**,
its **recent `skillsys(<id>)` activity** (last lessons / how long since the last edit), its **open threads**,
and a one-line **call**: *steady* · *has an unresolved thread* · *stale/quiet — may need a look* · *recurring
flaw — candidate for an OPTIMIZE pass*. If a system has no map or no criteria yet, say so — that is a DEFINE
gap (`init.md`), not a status the log can show.

## Scope fence (OBSERVE never crosses these)
- **No edits, no commits, no reruns.** OBSERVE drives no change to any skill, workflow, map, or registry.
  Updating status text is not OBSERVE's job either — the open-threads block is regenerated by CONSOLIDATE.
- **Do not judge an artifact's quality here.** That is OPTIMIZE's verify step (the criteria fixture + the
  human eye). OBSERVE reports *what the log and registry say*, not a fresh verdict on a product.
- **Do not open a system's full map/criteria to answer status** — the index + git pointers answer it. Open the
  map only when you have decided to act, i.e. you have already left OBSERVE for OPTIMIZE.
- **Surfacing → handoff, not action.** When OBSERVE reveals a flaw, recurring class, or a stale system worth
  fixing, name it and hand to OPTIMIZE — never start editing inside an OBSERVE pass.
